# TuiBox v0.1 Design

## Product

TuiBox is a lightweight terminal client for Linux and macOS. It imports proxy subscriptions, normalizes common endpoint formats, measures reachability, and connects through a separately distributed sing-box runtime. The repository, interface, logs, and documentation use English. TuiBox is licensed under GPL-3.0-only.

The first release supports VLESS, VMess, Trojan, Shadowsocks, Hysteria2, and TUIC. It accepts share-link lists, Base64 subscriptions, Clash YAML, and sing-box JSON. Users can choose TUN or loopback proxy mode and Global, Rule, or Direct routing. Server selection can be manual or based on a parallel TCP latency test.

## Architecture

TuiBox ships two Go binaries. `tuibox` is an unprivileged Cobra CLI with a Bubble Tea interface. It owns provider fetching, parsing, endpoint cache, settings, latency tests, telemetry consent, update checks, and user-facing errors. `tuiboxd` is a small system daemon installed through systemd or launchd. It owns connection lifecycle and is the only TuiBox component that runs with elevated privileges.

The binaries communicate through versioned, newline-delimited JSON over a local Unix socket. The daemon authenticates the kernel-reported peer UID against an installer-provided allowlist. The socket is root-owned and mode 0660. Requests are size-bounded and time-bounded. The API accepts only connect, disconnect, status, and health operations. It has no fields for shell commands, arbitrary arguments, executable paths, environment variables, or imported config paths.

sing-box remains a separate, pinned runtime. The daemon maps a validated TuiBox endpoint into a generated sing-box config, writes it atomically with mode 0600, runs `sing-box check`, and starts the process. TUN sessions run with required privileges. Proxy sessions drop to the caller's UID and GID and bind to loopback only.

## Components

### Domain and validation

The domain package defines protocol, route, mode, endpoint, TLS, transport, subscription, and connection status types. Validation is independent of the CLI, storage, parser, and daemon. Both the client and daemon call the same validation code. Strings and collections have explicit bounds. Hosts, ports, UUIDs, credentials, transport names, and protocol combinations fail closed.

### Subscription pipeline

The fetcher accepts HTTPS, limits redirects, applies a timeout, and caps response size. The detector tries explicit share links, whole-document Base64, JSON, and YAML without executing provider configuration. Format adapters map provider fields into the normalized endpoint model and return warnings for unsupported entries. The refresh service commits a new endpoint cache only when parsing produces at least one valid endpoint; otherwise, it keeps the last known-good cache.

### Secret and state storage

Subscription URLs are stored in macOS Keychain or Linux Secret Service. TuiBox invokes platform commands directly, never through a shell. If no credential service is available, it uses a mode-0600 fallback file and reports the downgrade. Cached endpoints and settings use atomic JSON files in a mode-0700 user data directory. These files still contain connection credentials and are treated as sensitive.

### Connection manager

The daemon admits one active session in v0.1. A replacement request first validates and checks the new config. Only after that succeeds does the daemon stop the old process and start the new one. It records non-sensitive status in memory. Disconnect sends a graceful signal, waits for a bounded interval, then kills the process if needed.

### TUI and CLI

The TUI shows connection state, route and connection modes, endpoint names, protocols, and latency. Keyboard commands add or refresh subscriptions, test endpoints, cycle modes, connect, disconnect, and quit. Cobra subcommands expose the same use cases for scripts and troubleshooting. The CLI never prints credentials or full subscription URLs.

### Updates and releases

GitHub Actions runs tests, vetting, formatting checks, and four cross-builds. Release jobs package `tuibox`, `tuiboxd`, service templates, licenses, and scripts, then publish SHA-256 checksums and GitHub provenance attestations. The installer and updater verify release metadata before replacing binaries. The core pin changes only through a reviewed compatibility update.

## Error handling

Errors cross boundaries as stable codes plus redacted human messages. Parser warnings identify the subscription and entry index, not credentials. Fetch failures preserve cached endpoints. RPC errors distinguish unavailable daemon, denied access, invalid request, core validation failure, and process failure. TUI errors remain visible until the next user action.

Logs use structured fields and an explicit redactor. Telemetry uses a separate allowlist rather than log forwarding. Strict opt-in is the default, and an empty build-time endpoint produces no network request.

## Testing

Parser tests use fixtures for every protocol and source format. Storage tests verify permissions and atomicity. RPC tests cover framing, size limits, peer authorization, timeouts, and state transitions. Config tests compare normalized JSON and run the pinned sing-box checker when available. CLI tests cover command wiring and redaction. Release checks cross-compile all supported targets and validate shell syntax.

Security review focuses on the privileged boundary: peer credentials, UID allowlisting, generated-only configs, process privilege dropping, fixed core path, bounded RPC, file permissions, update integrity, and secret redaction.
