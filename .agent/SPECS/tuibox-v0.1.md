# TuiBox v0.1 Spec

## Purpose

TuiBox is a lightweight, open-source terminal client for subscription-based proxy services. It lets Linux and macOS users import common subscription formats, inspect and test endpoints, select a server, and route traffic through a managed sing-box runtime.

## Non-goals

- Implement proxy protocols or cryptography.
- Execute provider-supplied sing-box or Clash configuration directly.
- Provide a GUI, mobile app, or Windows support.
- Provide custom rule editing, load balancing, remote administration, or multi-user session orchestration.
- Guarantee compatibility with provider-specific fields that do not map to the normalized endpoint model.
- Enable telemetry by default.

## User interfaces

### Primary TUI

Running `tuibox` opens a server list. The user can:

- move with arrow keys or `j`/`k`;
- add a subscription;
- refresh subscriptions;
- run latency checks;
- select manual or Auto Best mode;
- switch between TUN and Proxy connection modes;
- switch among Global, Rule, and Direct routing;
- connect or disconnect;
- inspect status and redacted errors;
- quit without leaving the terminal in a broken state.

### CLI

The CLI exposes scriptable commands:

```text
tuibox subscription add <name> <url>
tuibox subscription list
tuibox subscription update [id]
tuibox subscription remove <id>
tuibox server list
tuibox server latency [id|--all]
tuibox connect <endpoint-id|auto> --mode tun|proxy --route global|rule|direct
tuibox disconnect
tuibox status
tuibox telemetry enable|disable|status
tuibox doctor
tuibox update [--check]
tuibox version
```

### Daemon RPC

`tuiboxd` accepts newline-delimited JSON over a Unix socket. Supported operations are `connect`, `disconnect`, `status`, and `health`. Requests and responses have a versioned schema. The daemon derives the caller UID from the socket and ignores identity fields supplied by the request body.

## Domain model

An endpoint contains a stable ID, subscription ID, display name, protocol, host, port, protocol credentials, TLS options, transport options, and protocol-specific options. Validation rejects unknown protocols, invalid ports, empty credentials, invalid UUIDs where required, control characters, unbounded strings, arbitrary file paths, and unsupported transports.

A subscription contains an ID, display name, secret reference, detected/import format, refresh interval, last refresh time, and last error. Cached endpoints live in a mode-0600 state file inside a mode-0700 user data directory.

## Subscription ingestion

- Fetch only `https://` URLs by default.
- Apply a 15-second timeout and a 10 MiB response limit.
- Reject HTTPS-to-HTTP redirect downgrades.
- Parse individual share links, newline-separated URI lists, whole-document Base64, Clash YAML `proxies`, and sing-box JSON `outbounds`.
- Accept VLESS, VMess, Trojan, Shadowsocks, Hysteria2, and TUIC.
- Deduplicate endpoints by normalized connection identity.
- Preserve the last valid cache if refresh fails.
- Return warnings for skipped entries without exposing credentials.

## Connection behavior

### TUN mode

The client sends a normalized endpoint and route mode to `tuiboxd`. The daemon authenticates the peer, validates the request again, generates a root-owned temporary config, runs `sing-box check`, then starts the process. It enables automatic routes through sing-box's TUN inbound.

### Proxy mode

The daemon starts sing-box with the caller's UID and GID. The mixed HTTP/SOCKS inbound listens on `127.0.0.1:2080` by default. It never listens on a public interface unless a future explicit setting adds that behavior.

### Routing

- Global: default traffic uses the selected endpoint; private networks remain direct to avoid breaking local access.
- Rule: private networks and configured direct rule sets bypass the endpoint; remaining traffic uses it.
- Direct: no proxy endpoint is required; traffic uses a direct outbound.

## Privilege boundary

- The Unix socket is root-owned, mode 0660, and assigned to the installing user's primary group.
- The daemon validates peer UID against an explicit allowlist.
- RPC has bounded message size and read/write deadlines.
- The daemon never passes user strings to a shell.
- The daemon never accepts arbitrary config paths, executable paths, environment variables, or command arguments over RPC.
- Runtime configs are generated by TuiBox and written atomically with mode 0600.
- Logs redact credentials, subscription URLs, query values, userinfo, and endpoint addresses where practical.

## Secret storage

Use macOS Keychain through `/usr/bin/security` and Linux Secret Service through `secret-tool` when available. Invoke tools directly without a shell. If unavailable, store secrets in a separate mode-0600 JSON file inside the mode-0700 config directory and surface a warning in `doctor` and the TUI.

## Telemetry

Telemetry is strict opt-in. The setting defaults to false. Events use an allowlist and contain only app version, OS, architecture, coarse command/event name, protocol family, mode, route mode, success flag, and coarse duration bucket. No stable installation identifier is required in v0.1. If no build-time telemetry endpoint is configured, the sink remains a no-op even when enabled.

## Updates and installation

`install.sh` detects Linux/macOS and amd64/arm64, downloads a GitHub Release archive, verifies its SHA-256 digest, installs `tuibox`, `tuiboxd`, a pinned sing-box runtime, and the appropriate systemd/launchd definition. It never executes data downloaded from a subscription. `tuibox update` checks GitHub Releases and delegates the privileged replacement to the installed updater after digest verification.

## Edge cases and failure modes

- Empty or malformed subscriptions produce a clear error and preserve old endpoints.
- Duplicate names do not overwrite endpoints because IDs are content-derived.
- A failed latency check marks the endpoint unavailable without deleting it.
- A failed `sing-box check` never replaces a working session.
- A failed new connection leaves the old session running until the replacement config passes validation.
- Daemon restart reports disconnected state.
- Missing core, missing daemon, denied socket access, and unsupported OS have actionable `doctor` output.
- Interrupted state writes do not corrupt the previous file.
- Subscription URLs and credentials never appear in normal errors or telemetry.

## Acceptance criteria

- `go test ./...`, `go vet ./...`, and `go build ./cmd/tuibox ./cmd/tuiboxd` pass.
- Linux and macOS builds compile for amd64 and arm64 with `CGO_ENABLED=0`.
- Parser fixtures cover all six protocols and all four input formats.
- Invalid and oversized input tests fail closed.
- Generated proxy and TUN configs pass `sing-box check` for the pinned core.
- Daemon tests prove unauthorized peers are rejected and arbitrary command/config injection is absent from the request model.
- Store tests verify directory mode 0700, file mode 0600, atomic replacement, and failed-refresh cache preservation.
- Redaction tests cover URI userinfo, query tokens, UUIDs, passwords, and subscription URLs.
- The installer passes `sh -n`; release workflows build four target combinations and publish checksums.
- README documents install, uninstall, security model, telemetry, supported formats, troubleshooting, and contribution workflow.

## Test plan

- Unit tests for parsers, endpoint validation, config generation, redaction, state storage, telemetry allowlisting, update digest verification, and RPC framing.
- Integration tests for subscription HTTP behavior with `httptest`, daemon socket RPC with a temporary socket, and sing-box config validation when the pinned binary is available.
- CLI smoke tests for help and version output.
- Shell syntax checks for install and uninstall scripts.
- Cross-compilation checks for all supported targets.
